## Root cause

The Worker code currently deployed to `ppworker` only matches `/event/...` (long path) and only handles events. Your app uses **short paths**: `/e/...`, `/p/...`, `/v/...`. So when Facebook hits `partypanther.net/e/wednesday-jazz-affair`:

1. The route fires and the Worker runs.
2. The regex `/^\/event\/([^/]+)/` does not match `/e/...`, so `eventMatch` is null.
3. The Worker falls through to the "proxy to Lovable" branch.
4. Lovable's origin returns the generic SPA HTML with default OG tags.

Bonus bug: the Supabase query selects column `image`, but your `events` table column is `image_url`. So even on `/event/...`, the image would be missing.

The `wednesday-jazz-affair` Facebook preview that worked earlier was generated by Facebook reading from a stale cache or a different code path (possibly an earlier test). It is not reproducible with the current deployed code.

## Fix

Replace the deployed Worker with a slim version that:
- Matches `/e/`, `/p/`, `/v/` (and optionally legacy `/event/`, `/promo/`, `/venue/`)
- Detects crawlers
- Delegates to the existing `og-meta` Supabase Edge Function (which already correctly handles all three types, slugs and IDs, base64 image fallback, and proper meta tag formatting)
- Proxies everything else to Lovable

This keeps the Worker tiny and centralizes OG logic in one place (`supabase/functions/og-meta/index.ts`) so future tweaks don't require redeploying the Worker.

### New Worker code (you will paste this into Cloudflare → Workers & Pages → ppworker → Edit code)

```js
const CRAWLER_RE = /facebookexternalhit|Facebot|Twitterbot|LinkedInBot|WhatsApp|Slackbot|Slack-ImgProxy|TelegramBot|Discordbot|Pinterest|redditbot|Applebot|SkypeUriPreview|vkShare|W3C_Validator|Iframely|Googlebot|bingbot|preview/i;

const TYPE_MAP = {
  e: "e", event: "e",
  p: "p", promo: "p",
  v: "v", venue: "v",
};

const OG_ENDPOINT = "https://qgttbaibhmzbmknjlghj.supabase.co/functions/v1/og-meta";

export default {
  async fetch(request) {
    const url = new URL(request.url);
    const ua = request.headers.get("User-Agent") || "";
    const isCrawler = CRAWLER_RE.test(ua);

    // Path: /{prefix}/{slug}
    const match = url.pathname.match(/^\/([^\/]+)\/([^\/?#]+)/);
    const prefix = match?.[1]?.toLowerCase();
    const slug = match?.[2];
    const type = prefix ? TYPE_MAP[prefix] : null;

    if (isCrawler && type && slug) {
      try {
        const ogUrl = `${OG_ENDPOINT}?type=${type}&slug=${encodeURIComponent(slug)}`;
        const ogRes = await fetch(ogUrl, { cf: { cacheTtl: 300 } });
        if (ogRes.ok) {
          const html = await ogRes.text();
          return new Response(html, {
            headers: {
              "content-type": "text/html; charset=utf-8",
              "cache-control": "public, max-age=300",
              "x-og-source": "ppworker",
            },
          });
        }
      } catch (_) {
        // fall through to origin
      }
    }

    // Everyone else → proxy to Lovable origin
    const originUrl = new URL(url);
    originUrl.hostname = "partypanther.lovable.app";
    const proxyReq = new Request(originUrl.toString(), request);
    proxyReq.headers.set("Host", "partypanther.lovable.app");
    return fetch(proxyReq);
  },
};
```

Notes:
- No `env.SUPABASE_ANON_KEY` needed — the `og-meta` function is public (`verify_jwt = false`).
- `x-og-source: ppworker` header lets us verify in tests that the response actually came from your Worker.
- Includes `Googlebot`/`bingbot`/`preview` from your old regex so search engines also get crawler treatment.
